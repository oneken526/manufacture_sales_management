# /css-unify — CSS クラス共通化

Blade ビューと `resources/css/app.css` を横断スキャンし、以下の3種類の問題を検出・修正する。

1. **統一問題 (A)**: カスタムクラスをセマンティッククラスへ書き換えるべき箇所
2. **抽出問題 (B)**: 複数ファイルで繰り返されるスタイル群を新規共通クラスとして定義すべき箇所
3. **未使用問題 (C)**: app.css に定義されているが Blade で使われていないクラス（全ファイルスキャン時のみ）

**前提:** セマンティッククラスとは `resources/css/app.css` に定義された共通クラスを指す（`btn-primary`・`form-control`・`badge-*` 等）。

---

## ステップ 1: app.css の読み込み

`resources/css/app.css` を読み込み、以下を把握する。

- 定義されているセマンティッククラス名の一覧
- 各クラスのプロパティ（後述の重複検出に使う）
- 使用されていそうなカスタムクラスの定義（`search-input`・`search-select` 等）

---

## ステップ 2: Blade ファイルのスキャン

`resources/views/` 以下の全 `.blade.php` を読み込む。
ただし `resources/views/components/` 内はスキャン対象から除外する。

引数でファイルパスが指定された場合はそのファイルのみをスキャンする。

各ファイルから以下を **2種類** 収集する。

**① クラス使用箇所（HTML 属性）**
- `class="..."` 属性に含まれる全クラス名
- クラスが使われているタグ種別（button・input・select・span・div 等）
- 行番号

**② `<style>` タグ内の CSS 定義**
- ファイル内に `<style>` タグが存在する場合、その中の全クラス定義を抽出する
- 収集する情報: クラス名・プロパティ一覧・定義元ファイル名
- これらを「ファイルローカル CSS 定義」として app.css の定義と同等に扱い、ステップ3の比較対象に加える

```
収集例 (products/index.blade.php の <style> タグより):
  .prd-header    → { display:flex; align-items:center; justify-content:space-between; margin-bottom:1.5rem }
  .submit-btn--primary → { background:#4f46e5; color:#fff }
  .prd-search__input   → { border:1px solid #cbd5e1; border-radius:0.375rem; padding:0.5rem 0.75rem; ... }
```

## ステップ 3: 問題の検出

### 問題 A — カスタムクラスをセマンティッククラスに統一すべき箇所

以下のいずれかに該当するクラスを検出する。

| 検出条件 | 例 | 推奨置き換え先 |
|---------|---|--------------|
| `app.css` にカスタム定義があり、セマンティッククラスと同等のスタイルを持つ | `.search-input` | `form-control` |
| セマンティッククラスと役割が同じだがクラス名が異なる | `.custom-btn-blue` | `btn-primary` |
| 同じタグに対して複数のカスタムクラスが混在している | button に `.my-btn` と `btn` が混在 | `btn-primary` に統一 |
| `<style>` タグ内にページ固有クラスが定義されており（**1ファイルのみ**）、app.css の既存クラスと **80%以上** 一致する | `.prd-header` | `page-header` |

> 注意: 同じクラス（または類似クラス）が複数ファイルの `<style>` タグに存在する場合は問題Aではなく問題Bで検出する。問題Aと問題Bの両方で同一クラスが重複登録されないよう、問題Aの検出は1ファイルのみに存在するクラスに限定する。

記録する情報: ファイル名・行番号・現在のクラス名・推奨置き換え先・判定根拠・一致率。

### 統一先の判断基準（優先順位順）

複数のクラスが競合する場合、以下の基準を上から順に適用して統一先を決定する。

**基準1（最優先）: `app.css` に定義されているか**

- `app.css` に定義されているクラス → プロジェクト公式クラスとして優先採用
- `<style>` タグ内や Blade ファイル内にのみ定義されているクラス → 置き換え候補

```
app.css: btn-primary ✅  →  <style>内: submit-btn--primary ❌ → btn-primary に統一
```

**基準2: 役割ベース（セマンティック）か見た目ベースか**

- 役割ベースのクラス名（`btn-primary`・`form-control`・`badge-secondary`）→ 残す
- 見た目・画面固有のクラス名（`submit-btn--primary`・`prd-search__input`）→ 置き換え候補

```
btn-primary（役割）✅  vs  submit-btn--primary（見た目+役割混在）❌
form-control（役割）✅  vs  prd-search__input（画面固有）❌
```

**基準3: スタイルの一致度**

- プロパティが完全一致 → 自動的に統一候補とする
- 微妙に異なる（色・余白の差異）→ レポートに「要確認」フラグを立て、ユーザーに確認してから統一する

```
要確認フラグの例:
⚠️ .prd-header と .page-header はほぼ同等ですが padding が異なります（prd: 1.5rem / page: 1rem）。統一しますか？
```

### 問題 B — 重複スタイルを新規共通クラスとして抽出すべき箇所

**検出対象:** 異なる2ファイル以上の `<style>` タグに、同一または類似のスタイル定義が存在するもの。1ファイルのみに存在するページ固有クラスは問題Bの対象外とする（ただし app.css に同等クラスがあれば問題Aで検出する）。

ステップ2で収集した「ファイルローカル CSS 定義」と app.css の定義を合わせて、以下の3ケースを検出する。

**ケース1: 同じクラス名が複数ファイルの `<style>` タグに登場し、app.css に未定義かつプロパティが 80% 以上一致する**

- 推奨対応: app.css に昇格して共通クラスとして定義し、各ファイルの `<style>` タグ定義を削除する

```
例: .action-bar が customers/index と warehouses/index 両方の <style> に定義されている
→ app.css に .action-bar を追加し、各ファイルの <style> 定義を削除
```

**ケース2: クラス名は異なるが似たスタイルが複数ファイルに存在し、app.css に対応クラスがない**

- 推奨対応: app.css に新しい共通クラスを定義し、各ファイルの個別定義と置き換える

```
例:
  .prd-header と .wh-header が 100% 一致 かつ app.css に対応クラスなし
  → app.css に .page-header として新規追加し、両ファイルの <style> 定義を削除
```

**ケース3: app.css に似たクラスが既にあるが、プロパティが完全には一致しない（要確認フラグ）**

- 推奨対応: 差異を明示して `⚠️ 要確認` フラグを立て、ユーザーに統一可否を確認してから対応する

```
例:
  .prd-header と .wh-header が 100% 一致 かつ app.css の .page-header と 65% 一致
  どちらに統一しますか？ → [app.css 側を修正] / [ローカルを修正] / [別クラスとして残す]
```

#### 検出は3段階で行う

1. **事前チェック**: クラス名の完全一致を確認 → app.css定義済みなら比較①②へ、未定義かつ80%以上一致ならケース1、未定義かつ80%未満なら命名衝突として記録
2. **比較①**: ローカルクラス同士のプロパティを比較 → 80%以上一致で統一候補を抽出
3. **比較②**: 統一候補を app.css と照合 → ケース2/ケース3/問題A に振り分け

**事前チェック: クラス名の完全一致確認（ケース1の検出）**

比較①の前に、異なるファイルのローカルクラスの中に**同一クラス名**が存在しないか確認する。該当する場合は以下の順で判定する。

- app.css に**定義済み**の場合 → 比較①→②に進み問題Aとして振り分ける
- app.css に**未定義**の場合 → プロパティの一致率をさらに確認する
  - **80% 以上一致** → ケース1として即座に検出し、比較①・②は実行しない
  - **80% 未満**（命名衝突）→ ⚠️ 命名衝突として記録し、比較①・②はスキップする

ケース1・命名衝突・app.css定義済みのいずれにも該当しない場合のみ、以下の比較①→②を実行する。

**比較① ファイルローカルクラス同士の照合（検出トリガー）**

異なるファイルのローカルクラス同士のプロパティを比較し、**80% 以上一致するペア**を「統一候補」として抽出する。
この比較は app.css とは無関係で、Blade ファイルの `<style>` タグ内定義を横断的に突き合わせる。

```
例:
  .prd-header (products/index <style>)  → { display:flex; align-items:center; justify-content:space-between; margin-bottom:1.5rem }
  .wh-header  (warehouses/index <style>) → { display:flex; align-items:center; justify-content:space-between; margin-bottom:1.5rem }
  → 一致率 100% → 統一候補として検出
```

**比較② app.css との照合（ケース分類）**

比較①で得た共通プロパティ（積集合）を app.css の全クラスと比較し、以下のケースに振り分ける。

**一致率の計算方法: app.css 基準**

```
一致率 = (app.css クラスのプロパティのうち、共通プロパティと一致する数)
       ÷ (app.css クラスの総プロパティ数)
```

app.css 側に余分なプロパティがある場合は一致率が下がる。
これにより「置き換えると見た目が変わるリスク」が一致率に反映される。

```
例:
  共通プロパティ:        display / align-items / justify-content  （3個）
  app.css .page-header: display / align-items / justify-content / padding  （4個）
  一致率 = 3 ÷ 4 = 75% → ケース3（⚠️ 要確認）

  差異として報告: app.css の padding が置き換え後に追加適用されます
```

| 検出条件 | 振り分け |
|---|---|
| 同一クラス名が2ファイル以上に存在 + app.css 未定義 + プロパティ 80% 以上一致 | **ケース1**（事前チェックで自動検出） |
| 同一クラス名が2ファイル以上に存在 + app.css 未定義 + プロパティ 80% 未満 | **命名衝突**（⚠️ 手動対応・セクションBに出力） |
| 比較②: app.css と 80% 以上一致するクラスがある | **問題A**（既存セマンティッククラスに統一・**セクションAに出力**） |
| 比較②: app.css と 50〜79% 一致するクラスがある | **ケース3**（⚠️ 要確認） |
| 比較②: 50% 未満 or 対応クラスなし | **ケース2**（app.css に新規追加） |

記録する情報: クラス名・定義元ファイル・プロパティ一覧・ローカル同士の一致率・app.css との一致率・ケース種別・推奨対応。

### 問題 C — 未使用クラス（app.css にあるが Blade で使われていない）

> **引数指定時はスキップ**: 問題 C の検出はファイルパス引数が指定されていない場合（全ファイルスキャン時）のみ実行する。引数が指定された場合は問題 C を一切検出・出力しない。

全ファイルスキャン時: `app.css` に定義されているが、`resources/views/` 以下のすべての Blade ファイル（components/ を除く）のいずれにも登場しないクラスを検出する。

AskUserQuestion ツールを使い、以下を質問する。

- **選択肢**: 候補ごとに1つのオプション（「# 番号: 内容の要約」）

- **質問**: 「`{ローカルクラス名一覧}` をまとめる新しい共通クラス名を決めてください（提案: `{AIが提案するクラス名}`）」
- **選択肢**: 提案クラス名をデフォルトとして表示し、自由入力も可とする

**ケース3が含まれる場合の追加確認:**

ケース3（⚠️ 要確認）の番号が選択された場合、そのエントリごとに AskUserQuestion で以下を確認する。

- **質問**: 「`{ローカルクラス名}` と app.css の `{app.cssクラス名}` の差異（{差異の内容}）についてどう対応しますか？」
- **選択肢**:
  - `app.css 側を修正`: app.css のプロパティをローカルに合わせて変更し、ローカル定義を削除する
  - `ローカルを修正`: ローカルの `<style>` 定義を削除し、class= 属性で app.css のクラスを使用する（app.css は変更しない）
  - `別クラスとして残す`: 統一せず、ローカル定義をそのまま維持する

## ステップ 6: 選択された変更を適用する

選択された番号の変更を順番に実施する。

### 6-1. Blade ファイルの書き換え（問題 A・B 対象）

対象ファイルを読み込み、クラス名を置き換える。

**class= 属性の置き換え（問題A・問題Bのケース2・問題Bで問題Aに振り分けられたもの・ケース3「ローカルを修正」）:**
- `class="..."` 内の対象クラス名だけを置き換える（他のクラスは残す）
- `class="{{ ... }}"` などの動的クラスは内容を確認してから置き換える
- インデントや改行は元のコードに合わせる

```blade
{{-- 変換前 --}}
<input type="text" name="search" class="search-input">

{{-- 変換後 --}}
<input type="text" name="search" class="form-control">
```

```blade
{{-- 変換前 --}}
<select name="category_id" class="search-select">

{{-- 変換後 --}}
<select name="category_id" class="form-control">
```

**`<style>` タグ定義の削除（問題 B ケース1・2 対象）:**
- 問題Bで app.css に移行したクラスの定義を各ファイルの `<style>` タグから削除する
- `<style>` タグ内の残りの定義がなくなった場合はタグごと削除する

**ケース3の対応（ステップ5の追加確認結果に応じて処理する）:**
- `app.css 側を修正` が選択された場合: ローカルの `<style>` 定義を削除する（app.css の変更は Step 6-2 手順4 で行う）
- `ローカルを修正` が選択された場合: ローカルの `<style>` 定義を削除し、class= 属性で app.css のクラスを使用する（app.css は変更しない）
- `別クラスとして残す` が選択された場合: 何も変更しない

### 6-2. app.css の更新

以下を順番に実施する。

**手順1. 不要クラスの削除（問題 A で統一されたクラスが app.css にある場合）**
- Blade 側で使われなくなったカスタムクラスの定義を app.css から削除する
- 削除前にそのクラスがスキャン対象外のファイルで使われていないか確認し、懸念がある場合はユーザーに確認する

**手順2. 新規共通クラスの追加（問題 B で新規定義が必要な場合）**
- 関連する既存クラスの近くに新しいクラスを追加する
- プロパティは Blade での使われ方から推定する
- コメントで追加理由を1行記載する（`/* css-unify: 複数画面共通 */`）

**手順3. 未使用クラスの削除（問題 C で削除候補が選択された場合）**
- app.css から対象クラスの定義を削除する
- 削除前にスキャン対象外のファイルで使われていないか確認し、懸念がある場合はユーザーに確認する

**手順4. ケース3「app.css 側を修正」の反映（該当する場合）**
- app.css の対象クラスのプロパティをローカル定義に合わせて変更する
- Step 6-1 でローカルの `<style>` 定義は削除済みなので、ここでは app.css の変更のみ行う
