# /impl-flow — 実装〜テスト確認フロー

`docs/spec/manufacturing-sales-mgmt/feature-specs/{フォルダ名}/` に対応する機能の
実装〜テスト確認の流れを1ステップずつ管理する。

---

## 0. 起動時の処理

### 0-1. 引数の確認

引数（フォルダ名）が指定されていない場合は `AskUserQuestion` で確認する。

- `Glob` で `docs/spec/manufacturing-sales-mgmt/feature-specs/` 配下のフォルダ一覧を取得して選択肢にする

### 0-2. 関連ファイルのパスを決定する

引数を `{feature}` として以下のパスを設定する。

| 変数 | パス |
|---|---|
| 仕様書フォルダ | `docs/spec/manufacturing-sales-mgmt/feature-specs/{feature}/` |
| 進捗ファイル | `docs/spec/manufacturing-sales-mgmt/feature-specs/{feature}/impl-flow-progress.md` |
| 実装ガイド | `docs/spec/manufacturing-sales-mgmt/feature-specs/{feature}/implementation-guide.md` |
| チェックリスト | `docs/spec/manufacturing-sales-mgmt/feature-specs/{feature}/` 配下の `*-checklist.md`（Glob で検索） |
| テストファイル | `tests/Feature/` 配下を Glob で検索（`{feature}` をPascalCaseに変換してフォルダ名に使う） |

> **PascalCase変換ルール**: `quotation-management` → `Quotation`、`customer-management` → `Customer` のように先頭語をPascalCaseにして `tests/Feature/{Pascal}/` を検索する。

### 0-3. 進捗ファイルの読み込みまたは初期作成

進捗ファイルが存在する場合: `Read` で読み込み、現在のフェーズを把握する。

進捗ファイルが存在しない場合: 以下のテンプレートで新規作成する。

```markdown
# impl-flow 進捗: {feature}

機能名: {feature}
開始日: {今日の日付}

## ステップ進捗

| Step | 内容 | 状態 | 完了日 | メモ |
|---|---|---|---|---|
| Step 0 | テストファイル確認 | ⬜ 未着手 | - | |
| Step 1 | Red確認 | ⬜ 未着手 | - | |
| Step 2 | 実装チェック | ⬜ 未着手 | - | |
| Step 3 | Green確認 | ⬜ 未着手 | - | |
| Step 4 | 全体テスト確認 | ⬜ 未着手 | - | |
| Step 5 | 手動確認 | ⬜ 未着手 | - | |

## 実行ログ
```

### 0-4. 次に実行するステップを特定する

進捗ファイルの状態列を上から読み、最初の「⬜ 未着手」または「❌ 失敗」のステップを実行対象にする。

全ステップが「✅ 完了」の場合は完了メッセージを表示して終了する。

---

## Step 0: テストファイル確認

**目的**: テストファイルが存在するかを確認し、なければ作成方法を案内する。

### 実行

`Glob` で `tests/Feature/{Pascal}/` 配下の `*Test.php` を検索する。

### 判定と対応

| 結果 | 対応 |
|---|---|
| テストファイルが1件以上存在する | ファイル名と件数を表示して「✅ 完了」に更新し、Step 1 へ案内する |
| テストファイルが存在しない | 以下の案内を表示して終了する |

テストファイルが存在しない場合の案内:

```
テストファイルが見つかりませんでした。

テストケースを先に作成してください。作成方法は2つあります。

【方法1】手動で作成する
  参考: docs/spec/.../implementation-guide.md の Step 1

【方法2】/tsumiki:tdd-red を使って自動生成する
  1. 仕様書フォルダ内に requirements.md と testcases.md を用意する
  2. /tsumiki:tdd-red を実行する

テストファイルを作成したら、再度 /impl-flow {feature} を実行してください。
```

### 進捗ファイル更新（完了時）

```
| Step 0 | テストファイル確認 | ✅ 完了 | {日付} | {ファイル名}（{N}件） |
```

---

## Step 1: Red確認

**目的**: 実装前にテストが正しく失敗していることを確認する。

### 実行

テストファイルを `Glob` で特定して以下を実行する。

```bash
php artisan test {テストファイルパス}
```

### 判定と対応

| テスト結果 | 判定 | 対応 |
|---|---|---|
| 全件失敗 | ✅ 正常（Red状態） | 進捗を「✅ 完了」に更新してStep 2を案内 |
| 一部通過 | ❌ 異常 | 通過しているテストケース名を表示し、原因（実装が既に存在する等）を調べて案内 |
| 全件通過 | ❌ 異常 | テストの期待値が誤っている可能性を報告し、テストコードを読んで原因を案内 |
| 実行エラー | ❌ 環境問題 | エラーメッセージから原因（クラス未存在・DB未接続等）を特定して修正方法を案内 |

### 進捗ファイル更新（成功時）

```
| Step 1 | Red確認 | ✅ 完了 | {日付} | 全{N}件失敗を確認 |
```

### 進捗ファイル更新（失敗時）

```
| Step 1 | Red確認 | ❌ 失敗 | - | {失敗の概要} |
```

---

## Step 2: 実装チェック

**目的**: テストを通すために必要なファイルが揃っているかを確認する。

### 実行

テストファイルを `Read` で読み込み、`use App\Models\xxx` と `get('/xxx')` からモデル名・ルートを抽出する。  
以下の順番で `Glob` または `Grep` を使って存在確認する。

| # | 確認対象 | 確認方法 |
|---|---|---|
| 1 | マイグレーション | `database/migrations/` を Glob で検索 |
| 2 | モデル | `app/Models/{Model}.php` の存在確認 |
| 3 | ファクトリ | `database/factories/{Model}Factory.php` の存在確認 |
| 4 | ルート定義 | `routes/web.php` を Read してルートを確認 |
| 5 | コントローラー | `app/Http/Controllers/{Model}Controller.php` の存在確認 |
| 6 | ビュー（index） | `resources/views/{feature}/index.blade.php` の存在確認 |

### 出力例

```
## Step 2: 実装チェック結果

| # | ファイル | 状態 |
|---|---|---|
| 1 | database/migrations/..._create_quotations_table.php | ❌ 未作成 |
| 2 | app/Models/Quotation.php | ❌ 未作成 |
...

未作成のファイルが X件あります。
実装ガイド: docs/spec/.../implementation-guide.md を参照して実装を進めてください。
```

### 進捗ファイル更新

```
| Step 2 | 実装チェック | ✅ 完了 | {日付} | 未作成{N}件を確認 |
```

---

## Step 3: Green確認

**目的**: 実装後にテストがすべて通ることを確認する。

### 実行

```bash
php artisan test {テストファイルパス}
```

### 判定と対応

| テスト結果 | 判定 | 対応 |
|---|---|---|
| 全件通過 | ✅ Green達成 | 進捗を「✅ 完了」に更新してStep 4を案内 |
| 一部失敗 | ❌ 未完了 | 失敗しているテストケース名とエラーメッセージを一覧表示する。エラーの内容から原因（orderByの欠落・JOINの未実装等）を特定して修正方針を案内する |
| 全件失敗 | ❌ 実装未完了 | Step 2 に戻って実装ファイルの存在を確認するよう案内する |

### 進捗ファイル更新（成功時）

```
| Step 3 | Green確認 | ✅ 完了 | {日付} | 全{N}件通過 |
```

---

## Step 4: 全体テスト確認

**目的**: 新しい実装が既存の機能を壊していないことを確認する。

### 実行

```bash
php artisan test
```

### 判定と対応

| テスト結果 | 判定 | 対応 |
|---|---|---|
| 全件通過 | ✅ 影響なし | 進捗を「✅ 完了」に更新してStep 5を案内 |
| 今回追加分以外が失敗 | ❌ 影響あり | 失敗しているテストファイル名を一覧表示する。失敗しているテストファイルを `Read` で読み込み、今回の実装との関係（ルート競合・モデルの変更等）を特定して修正方針を案内する |

### 進捗ファイル更新（成功時）

```
| Step 4 | 全体テスト確認 | ✅ 完了 | {日付} | 既存テストへの影響なし |
```

---

## Step 5: 手動確認

**目的**: 自動テストでは確認できないバッジ色・レイアウト・操作感をブラウザで確認する。

### 実行

1. チェックリストファイルを `Read` で読み込む
2. 以下の案内を表示する

```
## Step 5: 手動確認

ブラウザで動作確認を行います。

### 準備
php artisan serve を実行してサーバーを起動してください。

### 確認チェックリスト
{チェックリストファイルのパス} を開いて、各項目を確認してください。

優先的に確認すべき項目:
{チェックリストから「手動確認」が必要な項目を抜粋して表示}

確認が完了したら「完了」と入力してください。
```

3. ユーザーが「完了」と入力するまで待機する
4. 「完了」が入力されたら進捗を更新して完了レポートを表示する

### 進捗ファイル更新（完了時）

```
| Step 5 | 手動確認 | ✅ 完了 | {日付} | ブラウザ確認済み |
```

---

## 実行ログの記録

各ステップの実行結果は進捗ファイルの「実行ログ」セクションに追記する。

```markdown
## 実行ログ

### {日付} Step 1 実行
- テストファイル: tests/Feature/Quotation/QuotationIndexTest.php
- 結果: 全24件失敗 → Red状態を確認

### {日付} Step 3 実行
- 結果: 20件通過 / 4件失敗
- 失敗内容: 「一覧は作成日の降順で表示される」→ orderBy未実装
- 修正方針: QuotationController@index に orderBy('created_at', 'desc') を追加
```
