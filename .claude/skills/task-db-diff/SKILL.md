# タスク仕様書 vs DB 設計 差異チェック

## 目的

TASK-xxxx.md に記載されたモデル定義と、実際の DB 設計（database-schema.sql およびマイグレーションファイル）を
照合し、差異を一覧化する。どちらが「正しい」かの判断は行わず、事実として差異を列挙する。

---

## 手順

### 1. タスク番号を特定する

ユーザーの指示からタスク番号を読み取る（例: 「タスク7」→ `TASK-0007`）。
番号が不明な場合はユーザーに確認する。

### 2. ファイルを読み込む

以下を並行して読み込む:

- **タスク仕様書**: `docs/tasks/manufacturing-sales-mgmt/TASK-{番号}.md`
- **DB 設計書**: `docs/design/manufacturing-sales-mgmt/database-schema.sql`
- **マイグレーション**: `database/migrations/` 内の関連ファイル（タスクで言及されるテーブル名で検索）

### 3. タスク仕様書からカラム情報を抽出する

タスク仕様書には、モデルの `$fillable` 配列やコードブロック内のカラム定義が含まれる。
以下を抽出する:

- **テーブル名**（モデルクラスから推定、または明示されている場合）
- **カラム名**（`$fillable` の各要素、またはコードブロック内の `$table->xxx()` の第1引数）
- **データ型**（記載がある場合）

### 4. DB 設計からカラム情報を抽出する

database-schema.sql と関連マイグレーションから以下を抽出する:

- **カラム名**
- **データ型**
- **NULL 可否**
- **COMMENT（論理名）**

マイグレーションが複数ある場合（初期作成 + ALTER TABLE）は、最終的な状態を正とする。

### 5. 差異を比較・分類する

以下の3種類に分類する:

| 種類 | 説明 |
|------|------|
| **名前の違い** | 同じ意味と思われるが、カラム名が異なる |
| **片方にのみ存在** | タスク仕様書にあって DB 設計にない、またはその逆 |
| **一致** | 両方に存在し、名前が同じ |

### 6. 結果を出力する

#### 出力フォーマット

各テーブルについて以下の形式で出力する:

```
## {テーブル名}（{モデル名}）

### 差異あり

| # | タスク仕様書のカラム | DB 設計のカラム | 種類 |
|---|---------------------|-----------------|------|
| 1 | `code`              | `product_code`  | 名前の違い |
| 2 | `unit`              | —               | 仕様書のみ |
| 3 | —                   | `deleted_at`    | DB設計のみ（softDeletes） |

### 一致

| カラム名 | 備考 |
|----------|------|
| `name`   |      |
| `notes`  |      |

### まとめ

- 差異: X件（名前の違い: N件、仕様書のみ: N件、DB設計のみ: N件）
- 一致: X件
```

#### 出力の注意点

- `deleted_at`・`created_at`・`updated_at` は Laravel の規約カラムのため、タスク仕様書に記載がなくても
  「DB設計のみ」として列挙せず、**備考欄に「Laravel規約」と記載して一致扱いにする**
- 外部キー（`_id` で終わるカラム）は名前全体で比較する（`category_id` ≠ `product_category_id`）
- 複数テーブルが対象の場合はテーブルごとにセクションを分ける
- 差異が1件もない場合は「差異なし（完全一致）」と明記する
