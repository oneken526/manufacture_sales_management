### REQ-001: ログイン（認証）

**信頼性:** 🔵　**出典:** PRD Phase 1

#### AC-001-01: 正常系 — 正しい認証情報でログイン成功

```
Given: 有効なメールアドレスとパスワードを持つユーザーが存在する
When:  ログインフォームに正しいメールアドレスとパスワードを入力して送信する
Then:  ダッシュボード画面にリダイレクトされる
       セッションが確立される
       ログイン済み状態になる
```

#### AC-001-02: 異常系 — 誤ったパスワードでログイン失敗

```
Given: 有効なメールアドレスを持つユーザーが存在する
When:  正しいメールアドレスと誤ったパスワードを入力して送信する
Then:  ログインページにとどまる
       「認証情報が一致しません」旨のエラーメッセージが表示される
       セッションは確立されない
```

#### AC-001-03: 異常系 — 存在しないメールアドレスでログイン失敗

```
Given: 入力したメールアドレスがDBに存在しない
When:  存在しないメールアドレスとパスワードを入力して送信する
Then:  ログインページにとどまる
       エラーメッセージが表示される（ユーザーの存在有無は漏洩しない）
```

#### AC-001-04: 境界値 — メールアドレス・パスワードが空欄

```
Given: ログインフォームが表示されている
When:  メールアドレスまたはパスワードを空のまま送信する
Then:  バリデーションエラーが表示される
       HTTPリクエストが認証処理まで到達しない
```

### REQ-002: ロール別アクセス制御

**信頼性:** 🔵　**出典:** PRD Phase 1 / ヒアリングQ10

#### AC-002-01: 正常系 — admin ロールは全ページにアクセス可能

```
Given: admin ロールを持つユーザーがログイン済みである
When:  管理者専用ページ（ユーザーマスタ管理等）にアクセスする
Then:  ページが正常に表示される
```

#### AC-002-02: 異常系 — sales ロールが admin 専用ページにアクセス

```
Given: sales ロールを持つユーザーがログイン済みである
When:  admin 専用ページのURLに直接アクセスする
Then:  403 Forbidden エラーまたはダッシュボードへのリダイレクトが発生する
       ページのコンテンツが表示されない
```

#### AC-002-03: 異常系 — 未ログインユーザーが保護ページにアクセス

```
Given: セッションが存在しない（未ログイン）状態である
When:  保護されたページのURLに直接アクセスする
Then:  ログインページにリダイレクトされる
```

#### AC-002-04: 正常系 — ロール別メニューの表示制御

```
Given: warehouse ロールを持つユーザーがログイン済みである
When:  ダッシュボードを表示する
Then:  倉庫・在庫関連メニューが表示される
       見積・受注の承認操作メニューは表示されない
```

### REQ-010: 得意先マスタ CRUD

**信頼性:** 🔵　**出典:** PRD Phase 1

#### AC-010-01: 正常系 — 得意先の新規登録

```
Given: admin または sales ロールのユーザーがログイン済みである
When:  社名・住所・締日・与信限度額・メールアドレスを入力して登録する
Then:  得意先が DB に保存される
       一覧画面に新しい得意先が表示される
       フラッシュメッセージで登録成功が通知される
```

#### AC-010-02: 正常系 — 得意先情報の更新

```
Given: 既存の得意先が存在する
When:  得意先の編集画面で情報を変更して保存する
Then:  変更内容が DB に反映される
       一覧画面に更新後の情報が表示される
```

#### AC-010-03: 正常系 — 得意先のソフトデリート

```
Given: 既存の得意先が存在し、紐づく受注・請求書が存在する
When:  得意先の削除操作を行う
Then:  ソフトデリート（deleted_at の設定）が実行される
       一覧画面から該当得意先が消える
       紐づく過去データ（受注・請求書等）は削除されない
```

#### AC-010-04: 異常系 — 必須項目が空の登録

```
Given: 得意先登録フォームが表示されている
When:  社名を空欄のまま送信する
Then:  バリデーションエラーが表示される
       DB にデータが保存されない
```

#### AC-010-05: 境界値 — 与信限度額に負の値を入力

```
Given: 得意先登録フォームが表示されている
When:  与信限度額に負の値（例: -1）を入力して送信する
Then:  バリデーションエラーが表示される
```

### REQ-020: 見積書作成・特別単価自動反映

**信頼性:** 🔵　**出典:** PRD Phase 2

#### AC-020-01: 正常系 — 特別単価が存在する得意先選択で自動反映

```
Given: 得意先Aの商品Xに対する特別単価（例: 8,000円）が登録されている
       標準単価は 10,000円 である
When:  見積書作成画面で得意先Aを選択し、商品Xを明細に追加する
Then:  明細の単価欄に特別単価 8,000円 が自動入力される
       標準単価は上書きされている
```

#### AC-020-02: 正常系 — 特別単価が存在しない得意先選択で標準単価反映

```
Given: 得意先Bに対する商品Yの特別単価が登録されていない
       商品Yの標準単価は 5,000円 である
When:  見積書作成画面で得意先Bを選択し、商品Yを明細に追加する
Then:  明細の単価欄に標準単価 5,000円 が自動入力される
```

#### AC-020-03: 正常系 — 複数明細行の登録

```
Given: 見積書作成画面が表示されている
When:  明細行を3件以上追加して登録する
Then:  すべての明細行が DB に保存される
       合計金額が各行の小計の合算として表示される
```

#### AC-020-04: 異常系 — 数量に0または負の値を入力

```
Given: 見積書の明細行に数量を入力している
When:  数量に 0 または負の値を入力して保存する
Then:  バリデーションエラーが表示される
```

### REQ-021: 見積承認ワークフロー（draft→pending→approved/rejected）

**信頼性:** 🔵　**出典:** PRD Phase 2 / ヒアリングQ1

#### AC-021-01: 正常系 — draft から pending への申請

```
Given: sales ロールのユーザーが draft 状態の見積書を保持している
When:  「承認申請」ボタンを押す
Then:  見積書のステータスが pending に変更される
       見積書の編集ボタンが非活性またはアクセス不可になる
```

#### AC-021-02: 正常系 — admin による承認（pending → approved）

```
Given: admin ロールのユーザーがログイン済みで、pending 状態の見積書が存在する
When:  承認画面で「承認」ボタンを押す
Then:  見積書のステータスが approved に変更される
       承認者・承認日時が DB に記録される
```

#### AC-021-03: 正常系 — admin による却下（pending → rejected）

```
Given: admin ロールのユーザーがログイン済みで、pending 状態の見積書が存在する
When:  承認画面で「却下」ボタンを押す
Then:  見積書のステータスが rejected に変更される
       却下理由のコメントが記録できる
```

#### AC-021-04: 異常系 — sales ロールが承認操作を実行しようとする

```
Given: sales ロールのユーザーがログイン済みである
When:  承認APIエンドポイントを直接呼び出す
Then:  403 Forbidden エラーが返却される
       見積書のステータスは変更されない
```

#### AC-021-05: 異常系 — pending 状態の見積書を編集しようとする

```
Given: pending 状態の見積書が存在する
When:  申請者が編集フォームにアクセスしようとする
Then:  アクセスが拒否されるか、フォームが読み取り専用で表示される
```

### REQ-022: 承認申請時にメール通知（承認者に送信）

**信頼性:** 🔵　**出典:** PRD Phase 2 / ヒアリングQ7

#### AC-022-01: 正常系 — 承認申請でメール送信

```
Given: admin ロールのユーザーが有効なメールアドレスを持っている
       MAIL_MAILER が設定されている
When:  sales ロールのユーザーが見積書を pending に変更する
Then:  admin ユーザーのメールアドレス宛にメールが送信される
       メールの件名に見積番号が含まれる
       メール本文に見積書詳細へのリンクが含まれる
```

#### AC-022-02: 異常系 — メール送信失敗時の見積書ステータス

```
Given: メールサーバーが接続不可の状態である
When:  見積書の承認申請を行う
Then:  見積書のステータスは pending に変更される（メール送信失敗でロールバックしない）
       エラーログにメール送信失敗が記録される
```

#### AC-023-01: 正常系 — 承認完了で申請者にメール送信

```
Given: 申請者が有効なメールアドレスを持っている

Then:  申請者のメールアドレス宛にメールが送信される
```

### REQ-024: 承認済み見積から受注変換

**信頼性:** 🔵　**出典:** PRD Phase 2

#### AC-024-01: 正常系 — approved 見積から受注変換

```
Given: approved 状態の見積書が存在する
When:  「受注変換」ボタンを押す
Then:  受注が新規作成される（ステータス: draft）
```

```
Given: pending 状態の見積書が存在する

Then:  操作が拒否される

```

#### AC-024-03: 異常系 — draft 状態の見積書から受注変換しようとする

```
Given: draft 状態の見積書が存在する
When:  受注変換操作を試みる
Then:  操作が拒否される
```

### REQ-031: 受注承認時に製造指示を自動発行（1受注=1製造指示）

**信頼性:** 🔵　**出典:** PRD Phase 2 / ヒアリングQ3

#### AC-031-01: 正常系 — 受注承認で製造指示が1件自動発行

```
Given: pending 状態の受注が存在する
When:  admin が受注を承認する
Then:  製造指示が1件自動的に作成される
       製造指示のステータスは pending となる
       製造指示に製番（製造番号）が自動採番される
       製造指示と受注が DB上で紐付けられる
```

#### AC-031-02: 境界値 — 既に製造指示が存在する受注を再承認しようとする

```
Given: 受注に対して既に製造指示が1件存在する
When:  何らかの操作で同一受注の承認処理が二重実行される
Then:  製造指示は追加作成されない（冪等性を保証する）
```

### REQ-032: 与信限度チェック（警告表示のみ）

**信頼性:** 🔵　**出典:** PRD Phase 2 / ヒアリングQ8

#### AC-032-01: 正常系 — 与信限度額を超えていない場合は警告なし

```
Given: 得意先Aの与信限度額は 1,000,000円 で、現在の未回収残高は 200,000円 である
When:  得意先Aへの受注（金額 300,000円）を登録・編集する
Then:  与信警告は表示されない
```

#### AC-032-02: 正常系 — 与信限度額を超えた場合に警告のみ表示

```
Given: 得意先Aの与信限度額は 1,000,000円 で、現在の未回収残高は 900,000円 である
When:  得意先Aへの受注（金額 200,000円）を登録または編集する
Then:  「与信限度額を超過しています」旨の警告メッセージが表示される
       受注の保存操作はブロックされない（警告のみ）
```

#### AC-032-03: 境界値 — 与信限度額と未回収残高がちょうど等しい場合

```
Given: 得意先Aの与信限度額は 1,000,000円 で、現在の未回収残高はちょうど 1,000,000円 である
When:  受注を1円でも追加する
Then:  与信警告が表示される
       受注は保存できる
```

### REQ-033: 受注残管理（受注数量 - 出荷数量）

**信頼性:** 🔵　**出典:** PRD Phase 2

#### AC-033-01: 正常系 — 出荷後に受注残が減算される

```
Given: 受注明細に商品X 100個の注文がある（出荷数量は 0）
When:  商品X の出荷を 60個分登録する
Then:  受注明細の受注残は 40個（100 - 60）と表示される
```

#### AC-033-02: 境界値 — 受注数量と等しい数量の出荷を登録

```
Given: 受注明細に商品X 100個の注文がある（既出荷 60個）
When:  残り 40個分の出荷を登録する
Then:  受注残は 0個 と表示される
       受注ステータスが「出荷済み」などに更新される（設計に従う）
```

#### AC-033-03: 異常系 — 受注残数量を超える出荷登録

```
Given: 受注明細に商品X の受注残が 40個 ある
When:  41個以上の出荷を登録しようとする
Then:  バリデーションエラーが表示される
       出荷は登録されない
```

### REQ-040: 製造指示の進捗ステータス更新

**信頼性:** 🔵　**出典:** PRD Phase 3

#### AC-040-01: 正常系 — pending → in_progress への更新

```
Given: manufacture ロールのユーザーがログイン済みで、pending 状態の製造指示が存在する
When:  製造指示の「製造開始」操作を行う
Then:  ステータスが in_progress に変更される
       変更日時が記録される
```

#### AC-040-02: 正常系 — in_progress → completed への更新

```
Given: in_progress 状態の製造指示が存在する
When:  製造指示の「製造完了」操作を行う
Then:  ステータスが completed に変更される
       完了日時が記録される
       在庫の入庫処理が自動実行される（REQ-041 連動）
```

#### AC-040-03: 正常系 — cancelled への更新

```
Given: pending または in_progress 状態の製造指示が存在する
When:  製造指示の「キャンセル」操作を行う
Then:  ステータスが cancelled に変更される
       キャンセル後は再開できない
```

#### AC-040-04: 異常系 — completed 状態からの再変更

```
Given: completed 状態の製造指示が存在する
When:  ステータスを pending に戻そうとする
Then:  操作が拒否される
       エラーメッセージが表示される
```

---

### REQ-041: 製造完了時の在庫入庫連動

**信頼性:** 🔵　**出典:** PRD Phase 3

#### AC-041-01: 正常系 — 製造完了で在庫が自動入庫される

```
Given: in_progress 状態の製造指示（商品X 50個）が存在する
       対象倉庫の商品Xの現在庫は 30個 である
When:  製造指示を completed に更新する
Then:  対象倉庫の商品Xの在庫が 30 + 50 = 80個 に増加する
       新しいロットが自動生成される（製番・製造日付き）
       在庫入出庫履歴に「製造入庫」の記録が残る
```

#### AC-041-02: 境界値 — 製造数量 0 の製造指示の完了

```
Given: 製造数量が 0 の製造指示が存在する
When:  completed に更新する
Then:  在庫数量は変化しない
       在庫入庫レコードは作成されないか 0 件の入庫として記録される
```

```
When:  製番トレース画面で MFG-2026-001 を検索する
```

### REQ-050: 出荷登録（ロット手動指定で引き当て）

**信頼性:** 🔵　**出典:** PRD Phase 3 / ヒアリングQ4

#### AC-050-01: 正常系 — ロットを手動指定して出荷登録

```
Given: 倉庫Aに商品Xのロット LOT-001（在庫 50個）と LOT-002（在庫 30個）が存在する
       受注に商品X 20個の未出荷残がある
When:  出荷登録画面でロット LOT-001 を指定して 20個の出荷を登録する
Then:  出荷レコードが作成される（ロット LOT-001 / 数量 20個）
       ロット LOT-001 の在庫が 50 - 20 = 30個 に減少する
       受注残が 20個 減少する
```

#### AC-050-02: 異常系 — 指定ロットの在庫数量を超える出荷

```
Given: ロット LOT-001 の在庫が 50個 である
When:  LOT-001 から 60個 の出荷を登録しようとする
Then:  バリデーションエラー「在庫数量を超えています」が表示される
       在庫は変化しない
```

#### AC-050-03: 正常系 — 複数ロットにまたがる出荷（分割引き当て）

```
Given: ロット LOT-001（30個）とロット LOT-002（30個）が存在する
When:  LOT-001 から 30個、LOT-002 から 20個 の合計 50個の出荷を登録する
Then:  出荷レコードが2件作成される
```
