# 見積書管理 機能仕様書

作成日: 2026-05-17  
対象システム: 製造業向け販売管理システム  
関連要件: REQ-013, REQ-014, REQ-015, REQ-018, REQ-020, REQ-021, REQ-022, REQ-025, REQ-026, REQ-046, EDGE-001

---

## 1. この機能の目的

得意先への見積書を作成・管理する機能です。  
営業担当者（sales）が見積書を作成し、管理者（admin）が承認することで正式な見積書となります。  
承認済みの見積書は受注に変換でき、以降の受注・製造・出荷・請求へとつながります。

---

## 2. 利用できるユーザー

| ロール | 操作できる内容 |
|---|---|
| 管理者（admin） | すべての操作が可能。承認・却下も行う |
| 営業（sales） | 作成・編集・申請・PDF出力・受注変換が可能 |
| 製造（manufacture） | 利用不可 |
| 倉庫（warehouse） | 利用不可 |

## 4. 見積書の登録項目

### 4-1. 基本情報

| 項目名 | 必須 | 入力内容 | 補足・制約 |
|---|---|---|---|
| 見積番号 | 自動 | 例: Q202605-001 | システムが自動採番する。手入力不可 |
| 得意先 | ○ | 登録済みの得意先から選択 | 選択後、特別単価が明細に自動反映される |
| 有効期限 | - | 見積書の有効期限（日付） | 期限を過ぎても操作は制限されない（表示のみ） |
| 作成者 | 自動 | ログイン中のユーザー | 自動セット。変更不可 |

> **見積番号の採番形式**  
> `Q{YYYYMM}-{3桁連番}` の形式で自動採番されます。  
> 例: 2026年5月に最初に作成した見積書は `Q202605-001`、次は `Q202605-002`。  
> 翌月になると連番は `001` からリセットされます。

### 4-2. 見積明細

1件の見積書に複数の明細行を登録できます。

| 項目名 | 必須 | 入力内容 | 補足・制約 |
|---|---|---|---|
| 商品 | ○ | 登録済みの商品から選択 | |
| 数量 | ○ | 数量（整数または小数） | 1以上の値を入力すること |
| 単価 | ○ | 販売単価（円） | 得意先選択時に自動セット。手動変更も可能 |
| 金額 | 自動 | 数量 × 単価 | 自動計算。入力不可 |

> **単価の自動反映（REQ-020）**  
> 得意先を選択すると、商品ごとに以下の優先順位で単価が自動セットされます。  
> 1. その得意先に設定された**特別単価**（登録されている場合）  
> 2. 商品マスタの**標準単価**（特別単価がない場合）  
>
> 単価は手動で上書きすることもできます。

## 5. 機能詳細

### 5-1. 新規登録

- 「新規登録」ボタンから登録フォームを開きます。
- 得意先を選択すると、明細の単価に特別単価が自動反映されます（Ajax）。
- 明細行は「行を追加」ボタンで増やせます。明細が0行の場合は保存できません。
- 保存すると **`draft`（下書き）** ステータスで作成されます。
- 見積番号はシステムが自動採番します。

### 5-2. 編集

- `draft`（下書き）状態の見積書のみ編集できます。
- `pending`（承認待ち）以降は**本人を含め誰も編集できません**（REQ-026）。
- 編集後に保存してもステータスは `draft` のまま変わりません。

### 5-3. 複製

- 既存の見積書を複製して新しい下書きを作成できます。
- 複製後は新しい見積番号が採番され、ステータスは `draft` になります。
- **却下された見積書を修正して再申請したい場合**に使います。

### 5-4. 承認申請

- `draft` の見積書から「承認申請」ボタンを操作します。
- 申請するとステータスが `draft` → `pending` に変わります。
- 申請と同時に admin ロールの全ユーザーにメール通知が送信されます。
- **メール送信が失敗した場合は申請処理ごとロールバックされます**。ステータスは `draft` のままになります（EDGE-001）。

### 5-5. 承認・却下（admin のみ）

- `pending` 状態の見積書を承認（`approved`）または却下（`rejected`）できます。
- 承認・却下時に申請者へメール通知が送られます。
- 却下時は**却下理由**を任意で入力できます。却下理由はメール通知にも含まれます。
- 却下された見積書を修正したい場合は、複製して新しい下書きを作成してください。

### 5-6. PDF出力

- 見積書詳細ページから見積書PDFをダウンロードできます（REQ-046）。
- PDF生成は開始時点のデータを使用します。生成中にデータが変更されても影響しません（EDGE-003）。
- PDFは日本語で出力されます（REQ-052）。

### 5-7. 受注変換

- `approved`（承認済み）の見積書のみ受注に変換できます（REQ-018）。
- 変換すると見積書の内容（得意先・明細）が受注にコピーされ、`draft` 状態で作成されます。
- 変換後も見積書はそのまま残ります（削除されません）。
- 変換後の受注は受注管理画面で操作します。

### 5-8. 削除

- 削除しても**データは完全には消えません**（論理削除）。  
  画面上は非表示になりますが、受注データとの紐付きは保持されます。

## 6. ステータスと承認フロー

見積書は以下の4つのステータスで管理されます。

| ステータス | 表示名 | 意味 |
|---|---|---|
| `draft` | 下書き | 作成直後の編集可能な状態 |
| `pending` | 承認待ち | 承認申請済みで管理者の判断を待っている状態 |
| `approved` | 承認済み | 管理者が承認した正式な状態 |
| `rejected` | 却下 | 管理者が却下した状態 |

```
[作成]
  ↓
draft（下書き）
  ↓ 承認申請
pending（承認待ち）
  ↓                 ↓
approved（承認済み） rejected（却下）
  ↓
受注変換が可能
```

> 承認フローの詳細なルール・メール通知の仕様は  
> [承認フロー 共通仕様書](../approval-flow.md) を参照してください。

---

## 7. 各ステータスでできる操作

| 操作 | draft | pending | approved | rejected |
|---|---|---|---|---|
| 内容を編集する | ○ | ✗ | ✗ | ✗ |
| 承認申請する | ○ | ✗ | ✗ | ✗ |
| 承認する（admin のみ） | ✗ | ○ | ✗ | ✗ |
| 却下する（admin のみ） | ✗ | ○ | ✗ | ✗ |
| 複製する | ○ | ○ | ○ | ○ |
| PDFをダウンロードする | ○ | ○ | ○ | ○ |
| 受注に変換する | ✗ | ✗ | ○ | ✗ |
| 削除する | ○ | ✗ | ✗ | ○ |

## 8. 他機能との連携

| 機能 | 連携内容 |
|---|---|
| 得意先管理 | 得意先選択時に特別単価を自動反映。得意先情報（締日・与信限度額）を見積書に紐付け |
| 商品管理 | 明細の商品選択時に標準単価を自動セット |
| 受注管理 | 承認済み見積書から受注を作成できる。変換元の見積書IDが受注に記録される |
| PDF出力 | barryvdh/laravel-dompdf を使用して見積書PDFを出力 |

## 9. エラーと注意事項

| 状況 | システムの動作 |
|---|---|
| 得意先が未選択で保存しようとした | バリデーションエラー。得意先を選択するよう案内 |
| 明細が0行で保存しようとした | バリデーションエラー。1行以上の明細を入力するよう案内 |
| 数量に1未満の値を入力した | バリデーションエラー。1以上の値を入力するよう案内（EDGE-006） |
| `pending` 状態の見積書を編集しようとした | 編集フォームを表示せず警告メッセージを表示（REQ-026） |
| 承認申請時にメール送信が失敗した | 申請処理ごとロールバック。ステータスは `draft` のまま（EDGE-001） |
| `approved` 以外の見積書で受注変換ボタンを押した | ボタン自体が表示されない（操作できない） |

## 10. 用語説明

| 用語 | 意味 |
|---|---|
| 見積書 | 得意先に対して商品の価格・数量・条件を提示する文書 |
| 見積番号 | 見積書を一意に識別する番号。形式は `Q{YYYYMM}-{3桁連番}`（例: Q202605-001） |
| 有効期限 | 見積書の有効期間の終了日。期限が過ぎても操作は制限されない |
| 特別単価 | 得意先ごとに設定する商品の個別単価。標準単価より優先される |
| 承認フロー | draft → pending → approved/rejected のステータス遷移を経る承認プロセス |
| 受注変換 | 承認済み見積書の内容をもとに受注を作成する操作 |
| 論理削除 | 画面上は削除されたように見えるが、データ自体はシステム内に保持される削除方式 |
