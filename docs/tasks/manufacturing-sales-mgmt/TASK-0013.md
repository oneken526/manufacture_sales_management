## 実装詳細

### 1. Quotation モデル 🔵
**信頼性**: 🔵 *DBスキーマ・要件定義より*

**実装ファイル**: `app/Models/Quotation.php`

- `SoftDeletes` トレイトを使用する
- `$fillable`:
  - `quotation_number`, `customer_id`, `user_id`, `valid_until`, `status`
  - `rejection_reason`, `submitted_at`, `approved_at`, `approved_by`
- リレーション:
  - `belongsTo(Customer::class)` - 得意先
  - `belongsTo(User::class, 'user_id')` - 作成者
  - `belongsTo(User::class, 'approved_by')` - 承認者（nullable）
  - `hasMany(QuotationDetail::class)` - 見積明細
- スコープ:
  - `scopePending(Builder $query): Builder` - `status = 'pending'` でフィルタ
  - `scopeApproved(Builder $query): Builder` - `status = 'approved'` でフィルタ
- ステータス値: `draft` / `pending` / `approved` / `rejected`

### 3. QuotationService 🔵
**信頼性**: 🔵 *要件定義（REQ-013〜016, REQ-025, REQ-026）より*

**実装ファイル**: `app/Services/QuotationService.php`

#### `create(array $data, User $user): Quotation` 🔵
- `NumberingService::generate('Q')` で `quotation_number` を採番する
- `quotations` と `quotation_details` をトランザクション内で保存する
- `status` 初期値: `draft`

#### `update(Quotation $quotation, array $data): Quotation` 🔵
- `draft` 状態のみ更新可能とする（REQ-026）
- `draft` 以外の場合は `\RuntimeException` または `\Illuminate\Auth\Access\AuthorizationException` をスローする
- 明細を一度 `deleteAll` してから再挿入するか、差分更新を行う

#### `submit(Quotation $quotation, User $requester): void` 🔵
**信頼性**: 🔵 *REQ-021・EDGE-001 より*
- `DB::transaction()` 内で以下を実行する:
  1. `status` を `pending` に更新し、`submitted_at` に現在日時をセットする
  2. admin ロールを持つ全ユーザー（`User::role('admin')->get()`）に `ApprovalRequestMail` を同期送信する
  3. メール送信例外発生時はトランザクションがロールバックされる（EDGE-001）
- `pending` 以外の状態から申請しようとした場合はエラーとする（二重申請防止）

#### `approve(Quotation $quotation, User $approver): void` 🔵
**信頼性**: 🔵 *REQ-015・REQ-022 より*
- `status` を `approved` に更新し、`approved_at` / `approved_by` をセットする
- 申請者（`$quotation->user`）に `ApprovalResultMail('approved', '')` を送信する

#### `reject(Quotation $quotation, User $approver, string $reason): void` 🔵
**信頼性**: 🔵 *REQ-015・REQ-022 より*
- `status` を `rejected` に更新し、`rejection_reason` を保存する
- 申請者に `ApprovalResultMail('rejected', $reason)` を送信する

#### `duplicate(Quotation $quotation, User $user): Quotation` 🔵
**信頼性**: 🔵 *REQ-013（見積複製）より*
- 元の見積の内容（得意先・有効期限・明細）をコピーして新しい `Quotation` を作成する
- 新しい `quotation_number` は `NumberingService::generate('Q')` で採番する
- 新しい見積の `status` は `draft` とする

## テスト要件（Feature テスト）

**テストファイル**: `tests/Feature/Services/QuotationServiceTest.php`

### テストケース1: 見積作成時の採番 🔵
**信頼性**: 🔵 *REQ-013・採番仕様より*
**Given**: quotations テーブルにレコードが存在しない
**When**: `QuotationService::create()` を呼び出す
**Then**: 作成された見積の `quotation_number` が `Q{YYYYMM}-001` 形式である

### テストケース2: draft→pending（submit）とメール送信 🔵
**信頼性**: 🔵 *REQ-021・EDGE-001 より*
**Given**: `draft` 状態の見積と申請者ユーザーが存在し、admin ユーザーが1名いる
**When**: `QuotationService::submit($quotation, $requester)` を呼び出す
**Then**: 見積の `status` が `pending` になり、`Mail::fake()` で `ApprovalRequestMail` が admin に送信されたことを確認できる

### テストケース3: pending→approved（approve）とメール送信 🔵
**信頼性**: 🔵 *REQ-015・REQ-022 より*
**Given**: `pending` 状態の見積と admin ユーザーが存在する
**When**: `QuotationService::approve($quotation, $admin)` を呼び出す
**Then**: `status` が `approved` になり、`approved_by` が設定され、申請者へ `ApprovalResultMail` が送信される

### テストケース4: pending→rejected（reject）と却下理由保存 🔵
**信頼性**: 🔵 *REQ-015・REQ-022 より*
**Given**: `pending` 状態の見積が存在する
**When**: `QuotationService::reject($quotation, $admin, '単価が不適切です')` を呼び出す
**Then**: `status` が `rejected` になり、`rejection_reason` に `単価が不適切です` が保存される

### テストケース5: pending 状態の見積を update → エラー 🔵
**信頼性**: 🔵 *REQ-026 より*
**Given**: `pending` 状態の見積が存在する
**When**: `QuotationService::update($quotation, $data)` を呼び出す
**Then**: 例外がスローされ、見積内容は変更されない

### テストケース6: duplicate で新しい draft 見積が作成される 🔵
**信頼性**: 🔵 *REQ-013 より*
**Given**: `approved` 状態の見積が存在する
**When**: `QuotationService::duplicate($quotation, $user)` を呼び出す
**Then**: 新しい `draft` 状態の見積が作成され、元の見積の得意先・明細がコピーされ、新しい採番がされている

## 注意事項
- PHP 8.3 の `declare(strict_types=1)` および型宣言を使用すること
- `QuotationService::submit()` のメール送信は必ず `DB::transaction()` 内で行い、EDGE-001 を遵守すること
- テストでは `Mail::fake()` を使用してメール送信をモックし、`Mail::assertSent()` で検証すること
- `SoftDeletes` を使用するため、`withTrashed()` が必要なクエリに注意すること
- `duplicate()` では `quotation_number` を新規採番すること（元の番号をコピーしないこと）
- ステータス遷移の検証ロジックは `QuotationService` 内に集約し、コントローラには書かないこと
